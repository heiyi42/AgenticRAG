1. 产品概述

AgenticRAG Chat 是一个面向课程知识库场景的 Web 端学习辅助系统，当前围绕三门固定课程构建： C语言、操作系统、网络安全实验。系统核心目标不是做开放域聊天，而是把“课程语料检索、学科化回答结构、 题目辅导、C 代码分析、多轮会话”整合成一个更适合教学场景的问答入口。

当前版本已经不再是单一的课程问答页面，而是形成了三条并行能力线： 普通课程问答、题目辅导与过程化解题、C 代码分析。用户在同一聊天界面中即可完成概念提问、步骤化做题和代码排错。

3. 产品定位与边界

产品定位：一个服务于课程教学和学习辅导的学科化智能助手，而不是开放域通用聊天产品。

  • 面向固定课程知识库，强调课程口径、章节结构、可教学性和稳定性。
  • 优先回答课程相关问题；对范围外问题，系统可走免检索直答，但不承诺课程知识库依据。
  • 网络安全实验内容仅面向课程教学、授权实验环境、靶场和受控演示环境，不面向真实未授权目标。
  • C 代码分析用于教学和排错辅助，不等价于任意代码执行平台或通用在线评测系统。

6. 核心功能说明

6.1 多轮会话与会话管理

  • 支持新建聊天、查看聊天、删除聊天、固定聊天、重命名聊天。
  • 支持会话标题自动生成，并通过事件流异步刷新标题。
  • 支持多轮上下文延续，后端具备摘要记忆与本地轮次裁剪能力。
  • 用户历史消息支持复制，以及在前端编辑后重新发送。
  • 回答以流式方式逐步展示，用户可在生成过程中停止回答。

说明：后端存在“清空会话”接口能力，但当前主前端界面主要暴露的是新建、固定、重命名、删除等操作。

6.2 学科选择

前端提供四个学科选项：自动、C语言、操作系统、网络安全实验。

  • 学科为“自动”时，系统根据问题内容决定是否单学科或跨学科处理。
  • 用户手动指定某一学科后，系统优先只使用该学科知识库，不主动扩展到其他学科。
  • 当前若用户手动将学科切换为“操作系统”，回答语言将切换为 English。
  • 当前学科选择主要作为前端当前聊天偏好使用，不是后端持久化的聊天字段。

6.3 Auto 模式与路由策略

Auto 模式当前采用“检索网关 + 学科路由 + 模式内编排”的组合策略。系统会先判断问题是否需要进入课程知识库， 再决定主学科、是否跨学科以及是否需要更深的检索链路。

  • 对范围外问题、寒暄和可直接回答的问题，系统可直接走免检索直答。
  • 对课程问题，系统会优先进入对应知识库，并按照学科回答骨架组织答案。
  • 当问题复杂或跨学科时，系统可在 Auto 内部升级为多学科整合或更深的回答链路。
  • Auto 当前强调“单学科优先、能直答则不强检索、能快答则不无故深搜”。

6.4 普通课程问答

普通课程问答仍是产品主链路，适合概念解释、机制说明、知识点比较、章节总结、实验原理说明等场景。 当前回答遵循“学科决定骨架，模式决定展开层级”的设计。

  • C语言：偏“结论、核心概念、代码示例、运行过程、易错点”。
  • 操作系统：偏“结论、本质、机制、图示化理解、对比、考试表达”。
  • 网络安全实验：偏“实验目标、实验环境、原理、步骤、结果观察、结论”。

6.5 题目辅导与过程化解题

题目辅导模块是当前版本的重要增强能力，主要解决“这道题怎么做、条件怎么抽取、过程怎么写、还能练什么题” 等解题型请求。当前版本以显式触发为主，即用户点击前端“题目辅导”按钮，或接口显式传入 problem_tutoring: true。

模块当前能力包括：

  • 题型识别：按三学科识别题目类型、目标、关键条件与知识点。
  • 模板化讲解：按题型组织“题型判断、条件抽取、过程说明、结论、易错点、类题方向”。
  • 题库检索：当前题库文件位于 data/tutoring_question_bank/questions.jsonl，共 300 道题，每科 100 道。
  • 类题推荐：返回同学科、同题型、相近知识点题目，作为继续练习入口。
  • few-shot 示例：系统会把相似题的解法摘要压入 prompt，提升解题回答稳定性。
  • 混合推荐：词面检索不足时，可回退到题库 embedding 索引进行补充推荐。

6.6 C 代码分析

C 代码分析模块用于教学排错和代码阅读。前端在学科选择为 C语言 时提供专用“代码分析”入口；此外， 后端也支持在普通输入框中根据代码块自动识别 C 代码分析请求。

  • 代码提取：从用户消息中提取 fenced code block 或高置信度 C 代码片段。
  • 编译检查：识别编译错误与警告。
  • 静态分析：在编译器支持时输出静态风险项。
  • 结构提取：在 clang 可用时提取 AST 结构；不支持时回退到轻量结构提取。
  • 受限执行：仅当编译成功且代码满足安全条件时，才尝试短时执行。

6.7 范围外问题处理

当前知识库只覆盖三类固定课程语料。对于超出范围的问题，系统会优先采用免检索直答，而不是强行从课程知识库中拼接答案。

  • 范围外问题不承诺课程知识库依据。
  • 范围外问题可返回通用说明，但不代表课程标准答案。
  • 如果用户需要课程口径结果，应手动选择对应学科并提出课程相关问题。

7. 推荐使用流程

7.1 普通问答

  1. 进入页面后直接新建聊天。
  2. 默认保持 Auto；若问题明确属于某一门课，可手动切换学科。
  3. 直接提问，系统会按流式方式返回答案。
  4. 若结果不够完整，可切换为 DeepSearch 重新提问。

7.2 题目辅导

  1. 先输入完整题干、关键条件、代码片段或实验现象。
  2. 点击“题目辅导”按钮，显式进入分步解题链。
  3. 等待系统输出题型、过程、结论、易错点和类题推荐。

7.3 C 代码分析

  1. 将学科切换到 C语言。
  2. 点击“代码分析”并粘贴完整 C 代码，或直接在主输入框中贴代码提问。
  3. 系统优先返回编译错误、警告、静态风险、结构信息和修改建议。
  4. 若满足执行条件，系统可附带受限执行结果。

8. 前端交互说明

  • 左侧侧边栏用于展示聊天列表，支持新建、固定、重命名、删除。
  • 顶部提供模式选择与学科选择。
  • 主区域展示消息流；回答支持 Markdown 渲染与流式输出。
  • 底部输入区提供普通发送、“题目辅导”按钮，以及在 C语言 学科下显示的“代码分析”按钮。
  • 系统通过 /api/events/chat-updates 推送标题更新事件，前端可在会话首次提问后异步刷新标题。

10. 关键请求参数与返回语义

  • mode：auto / instant / deepsearch。
  • subjects：显式学科数组，例如 ["C_program"]。
  • code_analysis：显式请求代码分析。
  • problem_tutoring 或 tutoring：显式请求题目辅导。
  • stream_chunk_size：流式分片大小。

返回事件中会包含 request_kind、retrieval_used、retrieval_gate_reason、 subject_route、assistant_meta 等元信息，用于前端提示当前回答所走链路。

12. 当前技术架构

  • 前端：单页 Web 界面，位于 templates/chat.html，负责会话交互、模式与学科切换、流式渲染。
  • 后端入口：webapp.py，基于 Flask 提供页面与 API。
  • 核心编排：webapp_core/chat_service.py 聚合路由、流式输出、检索支持、Auto 编排、题目辅导与代码分析能力。
  • 会话层：webapp_core/session_store.py 负责会话状态、本地持久化与摘要记忆快照。
  • 题目辅导：webapp_core/problem_tutoring_service.py 负责题型识别、题库推荐、规则求解与过程化提示构造。
  • 代码分析：webapp_core/code_analysis_service.py 负责 C 代码提取、编译检查、静态分析、结构提取与受限执行。
  • 知识库层：底层使用分学科 working_dir 维护索引与检索状态。

12.1 Agent 架构图

统一 Web 编排层负责请求分流与会话管理，底层 LightRAG 检索内核负责分学科检索与证据组织。

13. 普通课程问答模块

13.1 模块定位

普通课程问答模块是系统默认主链路，对应“直接提问并获得课程化回答”的使用方式。它同时承担三类任务： 判断是否应该进入课程知识库、识别属于哪个学科或是否跨学科，以及在 Auto / Instant / DeepSearch 三种模式下选择合适的检索与回答策略。

13.2 核心组件

组件 / 文件 职责
webapp_core/chat_streaming.py 消息主入口，解析请求参数，建立 SSE 事件流，决定先走哪条支路。
webapp_core/chat_routing.py 负责检索网关、学科路由、显式学科锁定和路由缓存。
webapp_core/chat_auto_orchestration.py 负责 Auto 模式预算划分、快速回答、充分性评审、第二学科补充和升级到 DeepSearch。
webapp_core/chat_retrieval_support.py 负责 Instant / DeepSearch / 跨学科合成 / 题目辅导支撑方法。
agenticRAG/instant_answer.py、agenticRAG/agentic_answer.py 底层统一 helper 入口，分别承载单跳检索和深度检索的非流式调用。
agenticRAG/agentic_runtime.py 维护 LightRAG 实例池、embedding 配置、tokenizer 和按学科 working_dir 取库逻辑。

13.3 入口判定与分流逻辑

  1. 前端把 message、mode、subjects 等字段提交到流式接口。
  2. 服务端先做高优先级识别：先看是否命中 C 代码分析，再看是否显式点击了题目辅导。
  3. 若都未命中，再判断是否属于寒暄/小型直答，最后才进入检索网关。
  4. 检索网关输出 need_retrieval、confidence、kb_relevance、direct_answerability。
  5. 若无需检索，则直接走免检索直答；若需要检索，则进入学科路由和模式编排。

13.4 学科路由与学科锁定

  • 若用户显式选择了学科，normalize_requested_subjects 会把别名统一成 C_program、operating_systems、cybersec_lab，随后直接锁定该学科。
  • 若用户未显式选择学科，系统先用关键词快路径做粗路由；置信度不足时，再调用结构化 LLM 路由器输出三学科分数。
  • 学科路由结果包含 primary_subject、cross_subject、confidence、ranked 等字段，供后续模式编排使用。
  • 当前若手动选择“操作系统”，上层会附加英文回答约束，因此底层仍查操作系统库，但最终答案语言切换为 English。

13.5 Auto / Instant / DeepSearch 的技术实现

这三种模式名称看起来像并列关系，但在当前实现里并不完全对称。DeepSearch 是一条多阶段深度检索链； Instant 是一条单跳检索链；Auto 则位于两者之上，负责“先快答、再评审、必要时升级”的编排。

13.5.1 Instant 链路

当前 Instant 相关实现主要分成“非流式 helper 入口”和“流式 helper 入口”两类：

  • 底层非流式 Instant helper：answer_instant() 固定使用 LightRAG 的 naive 模式做单跳查询，适合 CLI 或非 SSE 场景。
  • 流式 Instant helper：answer_instant_stream() 固定使用 LightRAG 的 naive 模式做单跳流式查询，返回 response_iterator 供 SSE 增量输出。
  • 当前主 Web 显式 Instant 与 Auto 内部 Instant 都复用了这条 answer_instant_stream() helper。

也就是说，当前 Instant 的差异点主要是“是否需要流式输出”，而不是 Web 和 Auto 各自维护一套不同的查询逻辑。

Instant 节点 / 函数 作用
answer_instant_stream 流式 Instant 包装器：固定以 mode="naive" 向 LightRAG 发起流式查询，返回 response_iterator 供 SSE 增量输出。当前 Web / Auto 的主流式 Instant 都走这一支。
_run_multi_subject_instant_stream Web 层单学科 Instant 包装器：负责学科样式提示、thread_id 学科隔离和 route 元数据回填。

显式 Instant 的典型执行顺序可以概括为：

显式 Instant
-> _stream_mode_with_retrieval(mode="instant")
-> _run_multi_subject_instant_stream
-> ask_instant_mode_stream
-> answer_instant_stream
-> LightRAG aquery_llm(mode="naive", stream=True)
-> 返回 answer / query_status / response_iterator

底层非流式 Instant helper 的执行顺序是：

Instant helper
-> LightRAG aquery_llm(mode="naive")
-> answer / elapsed_ms / query_status

13.5.2 Auto 链路

Auto 不是独立知识库查询器，而是“上层编排器”。它基于学科路由结果，决定应该先走快速链还是直接深搜， 以及在快速链返回之后是否需要评审、补第二学科或升级到 DeepSearch。

Auto 节点 / 函数 作用
_plan_auto_route 根据学科路由结果生成 Auto 执行计划，输出 auto_subjects、deep_subjects、complexity、route_chain、route_policy 等字段。
_start_speculative_auto_secondary_query 若存在第二学科且配置允许，则后台投机性启动第二学科 Instant 查询，为后续补充整合争取时间。
_ask_instant_stream 执行 Auto 的主学科快速查询。它走的是 answer_instant_stream()，固定使用 naive 模式查询。
_instant_heuristic_assessment 对 Instant 结果做第一层启发式充分性判断，例如是否为空、是否像失败信息、长度是否足够。
_should_review_instant_answer / _review_instant_answer 决定是否触发 LLM 评审，并由结构化 LLM 判断回答是否足以覆盖用户问题。
_run_multi_subject_auto_instant_stream 当主学科结果不足且存在第二学科时，取第二学科结果并做跨学科整合与充分性复评。
_run_auto_deep_stream 在预算允许时升级到 DeepSearch，并控制 deep 预算上限。
_stream_auto_mode Auto 总控函数，串联计划、主链查询、评审、补充整合和升级逻辑。

Auto 的典型执行顺序可以概括为：

Auto
-> _plan_auto_route
-> 判断 route_chain 是 instant 还是 deep
-> 若 instant:
   -> _start_speculative_auto_secondary_query（可选）
   -> _ask_instant_stream（主学科）
   -> _instant_heuristic_assessment
   -> _should_review_instant_answer / _review_instant_answer
   -> 若不足且有第二学科:
      -> _run_multi_subject_auto_instant_stream
   -> 若仍不足且预算足够:
      -> _run_auto_deep_stream
-> 若 deep:
   -> _run_auto_deep_stream
-> 返回 mode_used / upgraded / upgrade_reason / route / instant_review

因此 Auto 更像“策略层 Agent”。它并不直接触碰 LightRAG，而是调用 Instant 或 DeepSearch 支路，并根据回答质量继续调度。

13.5.3 DeepSearch 链路与节点功能

DeepSearch 是当前最完整的一条检索链。它先把问题编排成状态，再反复执行“规划 - 检索 - 证据评估 - 重试”，最后才生成最终回答。 这一链路主要由 chat_retrieval_support.py 和 agenticRAG/agentic_nodes.py 协同完成。

Web 层 DeepSearch 执行顺序如下：

DeepSearch
-> _stream_mode_with_retrieval(mode="deepsearch")
-> _run_multi_subject_deep_stream
-> _run_deepsearch_plan_state
-> build_global_subquestion_plan
-> route_subquestion_subjects
-> build_subquery_tasks
-> query_subquestion_tasks
-> judge_subquestion_results
-> prepare_subquestion_retry_plan（若需要）
-> build_final_answer_prompt
-> _stream_llm_text 输出最终答案

该链路中的关键节点职责如下：

  • build_global_subquestion_plan：先把整题拆成 1 到 5 个全局子问题，此时还不绑定课程知识库。
  • route_subquestion_subjects：对子问题逐个做学科路由，得到主学科和候选学科列表。
  • build_subquery_tasks：把“子问题”展开成可执行的“子问题-学科”检索任务，并附上 mode、top_k、chunk_top_k。
  • query_subquestion_tasks：并发执行所有检索任务，产出带 query_status、query_message、elapsed_ms 的结果表。
  • judge_subquestion_results：先按子问题聚合同源证据，再做规则和 LLM 充分性评估，判断当前证据是否足够回答该子问题。
  • prepare_subquestion_retry_plan：只对证据不足的子问题做改写、放大召回，必要时扩大学科范围后再重试。
  • build_final_answer_prompt：把所有子问题、检索状态、证据充分性、失败原因和结果证据编织成最终回答提示词。

Web 层当前使用的是“公共检索内核 + 自行流式收口”的 DeepSearch 编排，CLI 与 Web 共用同一套核心检索状态机。 状态核心字段定义在 agenticRAG/agentic_schema.py，包括 sub_questions、subquery_tasks、subquery_results、 needs_retry、insufficient_subquestion_ids、final_answer 等。

13.6 跨学科整合与输出

  • 当问题跨学科或主学科回答不足时，系统可以并行查多个学科，再做统一合成。
  • 合成阶段不是简单拼接，而是通过专门的“跨学科答案整合器”去重、分层，并做一次充分性评审。
  • 最终输出的 SSE 事件会带上 request_kind、retrieval_used、subject_route、assistant_meta 等元信息。
  • 回答结束后，消息会落入 SessionStore，并异步触发标题生成与更新事件。

14. 题目辅导模块

14.1 模块定位

题目辅导模块解决的是“解题型输入”，不是一般知识点解释。它的目标不是只给一个答案，而是把题型判断、条件抽取、 过程推导、规则求解、类题推荐和易错点讲解组织成完整的学习输出。

当前前端主链路以显式触发为准，即点击“题目辅导”按钮后，服务端才会在 _match_problem_tutoring_request 中接受该请求。仓库里的 tests/test_chat_streaming.py 也专门覆盖了“未显式点击时不进入题目辅导链”的约束。

14.2 核心执行流程

  1. match_request：先判断输入是否像题目，再用规则法生成首个分析结果。
  2. analyze_question：在规则结果基础上，可选地再调用结构化 LLM 做题型、目标、条件、知识点补全。
  3. select_template：从模板库中选出匹配题型，例如 c_output、os_page_replacement、os_banker、cybersec_lab_steps。
  4. recommend_similar_questions_hybrid：先做词面召回，必要时叠加 embedding 索引召回相似题。
  5. build_few_shot_examples：从相似题中提取标准解法与常见错误，作为 few-shot 示例。
  6. solve_deterministic：若题型属于已覆盖规则求解器，则先算出确定性结果。
  7. retrieve_knowledge：到当前学科知识库中补查“规则、公式、步骤依据、定义”。
  8. build_learning_outline 与 build_final_prompt：整理中间结构并交给 LLM 输出最终教学回答。

14.3 题型识别策略

  • 规则识别负责快速判定学科、题型、目标和条件，适合页面置换、进程调度、银行家算法、C 输出题等高特征输入。
  • 当启用结构化 LLM 解析器时，系统会在规则结果上补充更完整的 knowledge_points、extracted_conditions 和 answer_focus。
  • 如果用户手动指定了学科，LLM 分析结果也会被强制校正到该学科，以保证用户锁定行为优先。
  • 因此题目辅导不是“纯规则”也不是“纯生成”，而是规则、结构化 LLM 和检索材料的混合链路。

14.4 中间数据结构

结构 用途
TutoringProblemAnalysis 保存是否题目、学科、题型、置信度、目标、关键条件、知识点和答案组织重点。
ProblemTemplate 定义题型模板，包括别名、知识关键词、建议步骤、输出栏目、规则求解提示。
solver_result 记录规则求解器状态、求解器名称、过程步骤和结果字典。
learning_outline 返回前端可展示的结构化摘要，包含题型标签、推荐步骤、常见错误、类题推荐和检索摘要。

14.5 题库、few-shot 与 embedding 混合推荐

  • 题库主文件是 data/tutoring_question_bank/questions.jsonl，embedding 索引是 questions.embedding_index.json。
  • 默认先做词面相似度召回；当 top 结果不足够强时，才触发 embedding 召回，避免所有请求都走 embedding 成本。
  • few-shot 示例不会原样照抄整题答案，而是压缩为“解题步骤、答案骨架、常见错误”供最终 prompt 使用。
  • 类题推荐返回的是题号、题干摘要、难度和得分，便于用户继续练习。

14.6 规则求解器设计

求解器 覆盖题型 返回内容
页面置换求解器 FIFO / LRU / OPT / Clock 缺页次数、命中次数、缺页率、逐步页框状态。
进程调度求解器 FCFS / SJF / SRTF / RR 时间轴、完成时间、周转时间、等待时间和平均指标。
银行家算法求解器 安全性检查 / Request 试分配 Need、Work、Finish、安全序列和是否可分配。
PV 同步结构求解器 生产者消费者 / 读者写者 / 前驱关系 信号量设计框架、伪代码结构和关键同步点。
Diffie-Hellman 求解器 DH 密钥交换实验题 公开值、共享密钥和关键模运算步骤。

如果 solver_result.status = success，最终提示词会要求 LLM 优先采用规则结果，不得改写数值结论。 这使得计算型题目更像“程序化求解 + 教学讲解”的组合，而不是纯生成式回答。

14.7 课程检索与最终提示词组织

  • retrieve_knowledge 会把题型、知识点和题干压缩成面向课程依据的查询，固定走当前学科知识库的 hybrid 查询。
  • 检索结果并不直接返回给用户，而是作为“课程依据材料”注入最终 Prompt。
  • 最终 Prompt 会同时看到：题型分析、模板、知识库依据、相似题、few-shot、规则结果和安全边界。
  • 因此题目辅导模块本质上是一个多源材料编排器，而不是单次知识库问答。

15. C 代码分析模块

15.1 模块定位

C 代码分析模块面向“教学排错”和“代码阅读”，强调把工具结果翻译成学生能看懂的解释。 它不是在线判题系统，也不是任意代码执行沙箱。当前前端把它约束在 C语言 学科下，并提供专门的“代码分析”按钮。

15.2 触发与代码提取

  • 显式触发：前端传入 code_analysis: true。
  • 隐式触发：当用户直接贴入 fenced C 代码块或高置信度 C 代码片段时，后端也可自动识别。
  • CodeAnalysisService.extract_c_code_candidate 支持三类提取策略：标准 fenced code、未闭合 fenced code、普通文本中的 C 代码片段。
  • 系统会过滤明显非 C 语言提示词，避免把 Python、JavaScript 等代码误送入 C 编译链。

15.3 工具链与分析步骤

  1. 检测编译器：按 clang、gcc、cc、MSVC cl 等顺序寻找可用编译器。
  2. 编译检查：输出编译错误与警告。
  3. 静态分析：clang 走 --analyze，gcc 走 -fanalyzer，MSVC 走 /analyze。
  4. 结构提取：clang 可直接导出 AST JSON；否则使用轻量回退方法从源码抽取函数、声明和基础结构。
  5. 受限执行：仅在编译成功且满足安全条件时，才构建可执行文件并运行。

15.4 安全执行边界

  • 代码过长、缺少 main、或命中阻断模式时，直接跳过执行。
  • 阻断模式包括：文件与目录操作、外部命令或进程调用、网络 API、交互式输入、潜在越权头文件。
  • 执行时使用临时目录，并把 HOME、TMPDIR、TMP、TEMP 指向该目录，减少副作用外溢。
  • 执行超时、输出长度和代码长度都受环境变量约束。

15.5 输出结构与教学解释

字段 含义
compile_errors 编译错误列表。
warnings 编译器警告。
risk_findings 静态分析风险项。
structure AST 或回退结构提取结果。
execution 执行资格、构建状态、运行结果、超时情况和阻断原因。
tool_runs 编译、静态分析、AST、构建、运行五类工具调用的公共摘要。

工具分析阶段产出的不是一段字符串，而是一份结构化结果。随后 chat_code_analysis.py 会把这份结构化 JSON 再包装成教学解释 Prompt， 要求明确区分“编译错误、编译警告、静态风险、代码结构、修改建议、是否真的运行过程序”。

15.6 异常与降级策略

  • 如果没有检测到可用编译器，系统会返回“环境缺少 C 编译器”的说明，而不是误判为用户代码错误。
  • 如果编译器不支持 AST，则自动回退到轻量结构分析，不会直接失效。
  • 如果代码满足静态分析但不满足执行条件，系统仍会输出编译/结构/风险分析，只是不执行程序。
  • 整个链路返回的 route.chain 会标记为 code_analysis，前端据此显示专用加载文案和消息元信息。

16. 安全边界与合规说明

  • 网络安全实验相关回答仅面向授权实验环境、课堂演示和教学靶场，不应用于真实未授权目标。
  • 代码分析模块不会默认执行所有用户代码，只有在满足受限执行条件时才短时运行。
  • 涉及扫描、漏洞、利用、命令执行等内容时，产品应默认以课程实验和防护分析语境输出，不应给出面向真实攻击的操作指南。

17. 当前版本局限

  • 题目辅导当前主要依赖显式触发，尚未把自动识别作为主产品入口。
  • C语言题目的确定性求解仍不完整，复杂代码输出与逻辑推导仍主要依赖 LLM 与代码分析协同。
  • 学科选择当前主要存在前端状态中，不是后端持久化聊天字段。
  • 当前主前端未完整暴露所有后端管理接口，例如“清空会话”主要体现在 API 层。
  • 产品能力边界仍严格限定在三门课程；开放域问题不应被描述为系统核心能力。

18. 验收口径建议

  • 普通课程问答应在三学科内稳定返回结构化回答，并体现学科化骨架。
  • Auto 模式对明显范围外问题应允许免检索直答，对课程问题应优先进入知识库。
  • 题目辅导在显式触发后，应输出题型、过程、结论与类题推荐；规则可解题型应优先采用规则结果。
  • C 代码分析在可用编译器环境下，应区分编译错误、警告、静态风险与执行结果；在编译器缺失环境下，应正确提示环境问题。
  • 会话标题、删除、重命名、固定、流式输出等交互应表现清晰且结果可见。

19. 后续演进建议

  • 将题目辅导从“显式触发优先”逐步扩展到“显式触发 + 高置信自动识别”。
  • 补充 C语言 小代码片段的确定性求解器，提高输出题稳定性。
  • 将学科选择与更多聊天偏好正式纳入后端会话持久化。
  • 在前端增加链路说明面板，展示当前命中的请求类型、学科模板、规则求解与检索网关结果。
